# Polish

优化技术类文章，分两阶段执行：先输出诊断报告，用户确认后输出完整优化版。

## 执行流程

```
读取文章 → 四维诊断 → 输出诊断报告 → 用户选择模式（逐条确认 / 全量修复） → 输出优化版 + 改动摘要
```

## 阶段一：诊断报告

读取目标文件，按四个维度逐项检查。每个问题标注行号和严重程度。

严重程度定义：

- **严重**：技术错误、明显 AI 腔调，必须修改
- **疑似**：可能有误，需要用户确认
- **建议**：可以更好，但当前也能接受
- **轻微**：小瑕疵，不影响阅读

### 维度一：技术正确性

交叉验证代码示例与文字描述是否一致，术语使用是否准确。发现疑似错误时标注为"疑似"并给出理由，不直接断言。

额外关注：
- **API 兼容性描述**：不能只说"X 是 Y 的替代方案"，需要验证是否真的是同一用途、是否真的跨浏览器、是否有前置条件（如安全上下文、特定浏览器版本）
- **阈值/时间窗口类参数**：数字太松会漏报，太紧会误报，需要结合实际场景判断合理性

检查范围覆盖前端基础、AI Agent、后端/全栈、计算机基础四大领域。详见 `references/tech-domains.md`。

### 维度二：逻辑连贯性

- 段落之间是否有逻辑跳跃（前面说 A，后面突然说 C，缺了 B）
- 概念引入顺序是否合理（用到的术语是否先定义了）
- 表格、图表与上下文文字是否呼应
- 结论是否从前面的论述中自然得出
- 章节之间的过渡是否自然

### 维度三：去 AI 味 / 加人味

识别并消除 AI 腔调，增加个人视角。具体的 AI 味特征和人味特征详见 `references/ai-voice.md`。

### 维度四：语言质量

- 句子是否太长（超过 60 字考虑拆分）
- 用词是否重复（连续三段都用"本质上"）
- 是否有错别字、标点错误
- 表格列宽是否均匀，内容是否对齐
- **破折号（——）必须逐个检查**：markdown 渲染后视觉效果不明显，优先改用句号断句或逗号。诊断报告中列出所有破折号位置及修改建议

### 诊断报告输出格式

```markdown
## 诊断报告

### 技术正确性

- [严重] 第 X 行：具体问题描述
- [疑似] 第 X 行：具体问题描述 + 理由

### 逻辑连贯性

- [建议] 第 X 行：具体问题描述

### AI 味

- [严重] 第 X 行：具体问题描述

### 语言质量

- [轻微] 第 X 行：具体问题描述
- [轻微] 第 X 行：破折号"——"改为逗号"，"
```

输出诊断报告后，提供两种修复模式供用户选择：

- **逐条确认**：每条修改先展示 diff，用户逐个确认后再应用
- **全量修复**：一次性应用所有修改，用户看最终结果

## 阶段二：输出优化版

### 逐条确认模式

按严重程度从高到低逐条展示修改：先说明问题，再展示修改前后的 diff，用户确认后应用。用户可跳过某条（输入"跳过"或"skip"）或随时切换为全量修复。

### 全量修复模式

一次性应用所有修改，直接输出完整优化文章（直接覆盖原文件或输出新文件，由用户决定）。

两种模式均需在输出前对优化版做一轮自检：

- **技术正确性**：优化过程中引入的改动是否引入了新的技术错误（比如改了代码但没验证、补充了描述但不准确）
- **原意保持**：技术修正是否改变了作者的原意（修正事实错误 OK，改变观点不行）
- **信息完整性**：去 AI 味时是否误删了有信息量的内容（套话该删，但不能把有用的观点也删了）

自检通过后，末尾附改动摘要：

```markdown
## 改动摘要

| 改动类型 | 数量 | 说明     |
| -------- | ---- | -------- |
| 技术修正 | N    | 具体说明 |
| 逻辑优化 | N    | 具体说明 |
| 去 AI 味 | N    | 具体说明 |
| 语言优化 | N    | 具体说明 |
```

## 约束

- 不改变文章的核心观点和立场，只优化表达
- 不添加作者没有表达过的观点
- 技术错误会修正，但会在诊断报告中标出让用户确认
- 保留作者的个人风格，只消除"非人"的部分
- 发现疑似技术错误时标注为"疑似"并给出理由，不直接断言

## 风格

- 第一人称视角（"我自己的理解是"、"我很欣赏"）
- 问题驱动（先抛问题再解答）
- 表格化知识（对比、分类、总结）
- Mermaid 可视化（流程图、架构图）
- 代码实操（真实可运行的代码示例）
- 比喻接地气（"外骨骼"、"万能底牌"）
- 段落短（每段一个观点，不堆砌）
