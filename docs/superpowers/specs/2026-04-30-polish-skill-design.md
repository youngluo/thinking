# Polish Skill 设计文档

## 概述

设计一个 `polish` skill，用于优化技术类文章的技术正确性、逻辑连贯性、去 AI 味加人味、语言质量。适用范围覆盖前端、AI Agent、全栈和计算机基础。

## 触发方式

- 命令：`/polish <file.md>`
- 自然语言：在对话中说"帮我润色这篇文章"

## 执行流程

```
阶段一：诊断报告
  ├── 技术正确性检查
  ├── 逻辑连贯性检查
  ├── AI 味检测
  └── 语言质量检查

阶段二：输出优化版
  ├── 用户确认诊断方向后，输出完整优化文章
  └── 附带改动摘要
```

## 四个检查维度

### 1. 技术正确性

检查范围：

| 领域 | 检查内容 |
|------|---------|
| 前端基础 | React/Vue 原理、浏览器机制（事件循环、渲染管线）、CSS 布局、构建工具（Vite/Webpack）、TypeScript |
| AI Agent | Harness 组件、Agentic Loop、上下文管理、权限模型、LLM 原理、工具系统 |
| 后端/全栈 | Node.js、数据库、缓存、消息队列、微服务、Docker/K8s、API 设计 |
| 计算机基础 | 设计模式、数据结构、算法、网络协议（HTTP/TCP/WebSocket）、操作系统 |

检查方式：
- 交叉验证代码示例与文字描述是否一致
- 术语使用是否准确
- 发现疑似错误时，标注为"疑似"并给出理由，让用户判断
- 不依赖固定知识库，基于通用 CS 知识判断

### 2. 逻辑连贯性

- 段落之间是否有逻辑跳跃（前面说 A，后面突然说 C，缺了 B）
- 概念引入顺序是否合理（用到的术语是否先定义了）
- 表格、图表与上下文文字是否呼应
- 结论是否从前面的论述中自然得出
- 章节之间的过渡是否自然

### 3. 去 AI 味 / 加人味

**AI 味特征（需要消除）**：
- "在当今快速发展的技术环境中"
- "值得注意的是"
- "总而言之"、"综上所述"（每段结尾都来一个）
- 万能总结句："这不仅...而且..."
- 过度使用"首先...其次...最后..."的机械结构
- 每段都有"小结"，像在写教材
- 无观点的废话、套话

**人味特征（需要增加）**：
- 第一人称观点："我觉得"、"我特别认同"
- 接地气的比喻："外骨骼"、"万能底牌"
- 不完美的表达：偶尔用口语化的句子
- 有立场：不只是罗列，要有自己的判断
- 问题驱动：先抛问题再解答，而非平铺直叙

### 4. 语言质量

- 句子是否太长（超过 60 字考虑拆分）
- 用词是否重复（连续三段都用"本质上"）
- 是否有错别字、标点错误
- 表格列宽是否均匀，内容是否对齐

## 输出格式

### 阶段一：诊断报告

```markdown
## 诊断报告

### 技术正确性
- [严重] 第 45 行：React 18 的 concurrent 模式描述有误，createRoot 不是同步的
- [疑似] 第 120 行：HTTP/2 多路复用的解释可能过于简化，建议补充说明

### 逻辑连贯性
- [建议] 第 80 行：从 Hooks 直接跳到权限控制，中间缺过渡
- [轻微] 第 150 行：表格与上下文文字不呼应

### AI 味
- [严重] 第 3 行："在当今..."是典型 AI 开头，建议删除
- [建议] 第 90 行："综上所述"可删除，直接给结论

### 语言质量
- [轻微] 第 65 行："本质上"在前 3 段出现了 4 次，建议换词
```

严重程度说明：
- **严重**：技术错误、明显 AI 腔调，必须修改
- **疑似**：可能有误，需要用户确认
- **建议**：可以更好，但当前也能接受
- **轻微**：小瑕疵，不影响阅读

### 阶段二：优化版文章

- 输出完整优化后的 markdown 文件
- 末尾附改动摘要：

```markdown
## 改动摘要

| 改动类型 | 数量 | 说明 |
|---------|------|------|
| 技术修正 | 2 | 修正 React concurrent 模式描述，补充 HTTP/2 说明 |
| 逻辑优化 | 2 | 增加过渡句，调整表格位置 |
| 去 AI 味 | 5 | 删除套话 3 处，增加个人观点 2 处 |
| 语言优化 | 3 | 拆分长句 1 处，替换重复用词 2 处 |
```

## 约束与边界

- **不做**：不改变文章的核心观点和立场，只优化表达
- **不做**：不添加作者没有表达过的观点
- **做**：技术错误会修正，但会在诊断报告中标出，让用户确认
- **做**：保留作者的个人风格，只消除"非人"的部分
- **做**：发现疑似技术错误时，标注为"疑似"并给出理由，不直接断言

## 参考风格

基于 `apps/docs/experiences/` 目录下的文档风格：
- 第一人称视角（"我自己的理解是"、"我很欣赏"）
- 问题驱动（先抛问题再解答）
- 表格化知识（对比、分类、总结）
- Mermaid 可视化（流程图、架构图）
- 代码实操（真实可运行的代码示例）
- 比喻接地气（"外骨骼"、"万能底牌"）
- 段落短（每段一个观点，不堆砌）
